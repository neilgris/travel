# 决策记录 (DECISIONS)

本文件以流水账方式记录项目中的关键决策与原因，避免日后反复纠结同一问题。
新决策追加到末尾，不删除历史。

---

## 2026-06-30 · D1：文档体系与同步约定

**背景**：项目希望沉淀文档，使每个新会话能"冷启动"快速理解项目，且后续每次改动文档与代码同步更新，保证不跑偏。

**决策**：建立以下文档，并约定同步纪律。

文档体系：
- `CLAUDE.md`（项目根，⭐ 新会话自动加载）：项目概览、技术栈、如何运行、目录结构、关键约定、指向其它文档的链接。
- `docs/specs/<日期>-travel-journal-design.md`：需求 & 设计文档（头脑风暴产物）。
- `DECISIONS.md`（本文件）：关键决策流水账。
- 实现计划 `docs/plans/<日期>-...-plan.md`：设计定稿后由 writing-plans 技能生成。
- `README.md`：可选，暂缓（私人项目，先靠 CLAUDE.md）。

**同步纪律**：每次改动，相关文档与代码一起更新。
- 数据模型变化 → 更新设计文档 + `CLAUDE.md`
- 新的取舍/选择 → 追加到本文件 `DECISIONS.md`
- 需求变化 → 更新设计文档

## 2026-06-30 · D2：项目定位与技术栈（讨论中，待设计定稿确认）

- 定位：**纯私人**旅游记录网站，无账号系统、无多用户。
- 运行：**本地单机网页**（不部署云端），未来可从本地升级到云端。
- 技术栈：Python + Flask + SQLite + Jinja2 模板（服务端渲染，极少 JS）。
- 存储：SQLite 单文件（要做花费统计，优于 JSON）。图片存本地 `uploads/`，库中存路径。
- 配图定位：**少量配图**，以文字记录为主。

## 2026-06-30 · D3：核心数据模型定稿

设计文档：`docs/specs/2026-06-30-travel-journal-design.md`。

- **同行人 Person 独立成表、可复用**（姓名+照片），与 Trip 多对多。原因：同一人会出现在多次旅行。
- **城市 City 独立成表、可复用**，含经纬度，为第二版地图展示做准备；坐标自动地理编码。
- **行程段 Leg 建模**：有序的「出发城市→到达城市 + 出行方式」。原因：用户的路线本质是分段、有顺序、每段有交通方式（如 北京→飞机→香港→游轮→冲绳…），trip 级的"出行方式多选"画不出路线，且以后补做迁移成本高，故现在一步到位。
  - 推论：Trip 的城市/出行方式由 Leg 推导，**砍掉 TripCity 关联表**。
  - Leg 不加日期字段，仅靠 seq 排序。
- **Leg 与 Day 不直接关联**，通过 City 间接相连。两层分离：Leg=怎么走，Day=每天干啥。原因：低耦合，地图与日常记录互不影响。
- **多币种**：每个旅程可声明多个外币，各自手动设汇率；**汇率方向 = 1 人民币 = ? 外币**；换算 `人民币 = 外币 ÷ 汇率`；不自动取汇率。
- **Entry 五类**：吃饭/游玩/购物/住宿/交通。原因：住宿、交通是旅行大开销，纳入才能完整统计。
- **同天记录按添加顺序**排列，不加时间字段、不做拖拽。YAGNI。
- **移动日**（游轮在途等）简单处理：自选一个城市，不设"在途"特殊类型。

## 2026-06-30 · D5：技术落地

第一版实现完成后的技术细节记录。

- **ORM**：Flask-SQLAlchemy（模型层），用 `db.get_or_404` 的 2.0 形式避免 LegacyAPIWarning。
- **地理编码**：OpenStreetMap Nominatim（免费无 API key），失败返回 None。
- **图表**：Chart.js（base.html CDN 引入），实时渲染统计数据。
- **金额**：`Numeric` 列 + Python `Decimal`，每条目四舍五入两位（ROUND_HALF_UP），各汇总口径一致。
- **时间戳**：`created_at` 用 timezone-aware `datetime.now(timezone.utc)`（避免 utcnow 弃用警告）。
- **Flash 消息**：需要 `SECRET_KEY`（已加入 config，env 可覆盖）。
- **币种下拉**：记录花费时只能选「CNY + 本旅程已声明外币」，to_cny 对未配置币种有防御性 0.00 兜底（UI 约束，正常不可达）。
- **编辑旅程路由**（最终审查）：`/<trip_id>/edit` 路由复用 `_apply_form` 与 `form.html`（已支持编辑模式）；`add_entry` 增加 `day.trip_id != trip_id` 完整性校验，不匹配时返回 404。

## 2026-06-30 · D6：数据库演进策略 & 文档更新标准

**背景**：补全文档体系，明确「冷启动 / AI 接手」需要的沉淀，以及各文档的更新规则；同时把一个隐藏坑（无 migration）定下处理方式。

### 数据库 schema 演进（无 migration）
- 现状：用 SQLAlchemy `create_all` 自动建表，**无 Alembic/migration**。
- 决策：**纯私人单机项目，改模型时直接删 `instance/travel.db` 重建**（数据是测试/个人数据，可承受丢失）。不引入 migration 框架（YAGNI）。
- 注意：一旦库里有想保留的真实旅行数据，再改 schema 前需**手动备份/迁移**；届时若频繁改模型，可重新评估引入 Alembic。

### 文档体系（沉淀什么 & 给谁看）
四类文档，更新规则**由类型决定，不是每份每次都改**：

| 文档 | 类型 | 更新规则 |
|------|------|----------|
| `docs/specs/*-design.md` | **状态快照** | 改数据模型/页面/统计/需求时同步到与现实一致；上线或移除**功能**时更新「功能总览（当前已实现）」节。状态型文档过时即是 bug。 |
| `CLAUDE.md` | **入口/地图** | 改项目结构（新增顶层目录/模块）、技术栈、运行方式时更新。含目录结构与约定，必须名副其实。 |
| `DECISIONS.md` | **追加日志** | 有新取舍 / 踩坑 / 推翻旧决策时**追加新条目**，不改旧条目（推翻则新条目里说明 supersede 了哪条）。 |
| `docs/plans/*` | **一次性产物** | 某版本的实施计划，做完即视为归档，**不再维护**。 |

### 不该碰文档的情况
- 改 bug、调样式、加测试、重命名内部变量 → **只靠 commit message 记录**，不更新文档。
- 强行「每次迭代都更新所有文档」会制造**假同步**，反而有害。

### 一次迭代的文档自检（合并前）
对照上表：改了模型/页面/统计/需求→动 spec；改了结构/运行→动 CLAUDE.md；有新决策→追加 DECISIONS；上线/删功能→改 spec 功能总览节；其余→不动。

### 暂不做
- `README.md`：仍暂缓（私人项目，CLAUDE.md 已覆盖入口职责）。
- `CHANGELOG.md`：不单独维护，git log + 规范 commit message（`feat/style/docs/fix` 前缀）已足够。
